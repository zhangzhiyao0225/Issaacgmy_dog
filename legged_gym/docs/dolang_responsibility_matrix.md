# Dolang 项目责任划分说明

本文档用于明确 dolang 项目中我方与合作方的工作边界。我方工作范围限定为策略 ONNX 文件导出、对应配置文件整理及接口说明，不承担合作方机器人硬件、部署平台、实机运行、安全验证或系统集成责任。

## 1. 角色定义

| 角色 | 定义 |
| --- | --- |
| 我方 | 负责训练结果导出、ONNX 文件、配置文件和必要接口说明的一方 |
| 合作方 | 负责机器人硬件、部署系统、实机控制、驱动适配、测试验证和最终应用的一方 |

## 2. 总体边界

我方负责：

1. 基于双方确认的机器人数据和训练配置导出 ONNX。
2. 提供与 ONNX 对应的配置文件。
3. 说明 ONNX 输入输出、观测维度、动作维度、动作顺序、默认关节角和关键配置参数。
4. 在交付范围内解释文件使用前提和版本对应关系。

合作方负责：

1. 提供并确认机器人 URDF、关节命名、关节顺序、限位、电机能力、默认站姿和部署端控制约定。
2. 负责 ONNX 在其平台上的加载、推理、加速、转换、封装和运行。
3. 负责部署端观测构造、状态估计、坐标系转换、动作下发和低层控制。
4. 负责实机安全、联调、测试、验收、上线和后续维护。

## 3. 责任矩阵

| 工作项 | 我方责任 | 合作方责任 | 备注 |
| --- | --- | --- | --- |
| 机器人机械参数提供 | 不负责 | 负责 | 包括站立高度、趴下高度、关节零位、默认站姿、限位等 |
| URDF/CAD 准确性 | 不负责 | 负责 | 我方仅按合作方提供资料使用 |
| 训练配置整理 | 负责 | 配合确认机械数据 | 配置以双方确认版本为准 |
| 默认关节角配置 | 负责写入配置 | 负责提供并确认数值 | 当前默认站姿为 `0, 0.75, -1.5` |
| ONNX 导出 | 负责 | 接收并验证文件完整性 | 导出版本需与配置版本对应 |
| ONNX 输入输出说明 | 负责 | 按说明接入部署端 | 包括 shape、dtype、观测动作顺序 |
| 部署端观测构造 | 不负责 | 负责 | 合作方需保证观测顺序、单位、坐标系一致 |
| 部署端动作映射 | 不负责 | 负责 | 合作方需保证 12 个动作与关节顺序一致 |
| 推理框架适配 | 不负责 | 负责 | 包括 TensorRT、OpenVINO、onnxruntime、端侧 SDK 等 |
| 电机驱动和低层控制 | 不负责 | 负责 | 包括 PD 控制、电流/力矩控制、限位保护等 |
| 实机测试和安全 | 不负责 | 负责 | 包括急停、防摔、场地安全、人员安全 |
| 性能验收 | 仅限交付文件一致性 | 负责实机效果验收 | 实机效果受硬件、控制器、状态估计和部署实现影响 |
| 问题排查 | 限 ONNX/配置解释 | 负责部署链路排查 | 若问题来自部署实现，由合作方处理 |
| 后续重训或重导出 | 需另行确认 | 提供新输入条件 | URDF、控制频率、观测或硬件变化均可能需要重新确认 |

## 4. 我方不承担的事项

除非另有书面约定，以下事项不属于我方责任：

1. 合作方机器人硬件、结构、电机、传感器、驱动器、控制板或通信链路问题。
2. 合作方部署程序、推理框架、操作系统、中间件、ROS/ROS2 节点或 SDK 接入问题。
3. 合作方修改 ONNX、配置、观测顺序、动作顺序、单位、坐标系、控制频率后产生的问题。
4. 实机运行中的跌倒、碰撞、损坏、人员安全、场地风险或生产使用风险。
5. 未经双方确认的 URDF 参数、动力学参数、电机参数或机械零位偏差导致的问题。
6. 模型在合作方真实环境中的速度、稳定性、鲁棒性、能耗或通过率保证。
7. 合作方将 ONNX 转换为其他格式后的精度、速度、兼容性或数值一致性问题。

## 5. 合作方需确认的输入条件

为保证 ONNX 与配置文件可被正确使用，合作方应确认以下信息：

| 项目 | 需确认内容 |
| --- | --- |
| URDF | 文件版本、关节名称、关节顺序、限位、坐标系 |
| 默认站姿 | 站立高度约 `0.47 m`，趴下高度约 `0.12 m`，关节角 `hip=0`, `thigh=0.75`, `calf=-1.5` |
| 控制周期 | 部署端控制频率需与配置约定一致或由合作方自行适配 |
| 观测输入 | 观测项顺序、单位、坐标系、归一化和历史帧处理 |
| 动作输出 | 12 个动作对应的关节顺序、动作缩放和限幅 |
| 安全策略 | 急停、限位、降级、失控保护和测试流程 |

## 6. 交付验收边界

我方交付验收以文件为准：

1. ONNX 文件可加载。
2. 配置文件与 ONNX 文件版本一致。
3. 交付说明中记录输入输出维度和关节顺序。
4. 文件摘要可校验。
5. 合作方确认收到交付包。

实机部署是否可运行、运行效果是否满足合作方场景、是否达到商业应用标准，均不作为我方文件交付验收条件。

## 7. 变更处理

以下情况应视为需求或输入条件变化：

1. 更换 URDF、CAD、机器人结构、电机、传感器或控制板。
2. 修改关节名称、关节顺序、限位、零位或默认站姿。
3. 修改观测输入、动作输出、控制频率、动作缩放或坐标系。
4. 将 ONNX 转换到其他推理格式或部署到不同硬件平台。
5. 要求新增恢复、上台阶、复杂地形、载重、抗扰或其他能力。

发生上述变化时，是否重新训练、重新导出、重新配置或新增交付物，应由双方另行确认。
